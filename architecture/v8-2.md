## Purpose

This document is the canonical full production architecture for Mycelis.

V8.2 is the canonical full production target for Mycelis and the active B2+ delivery frame. It extends the V8.1 living-organization baseline into a distributed, learning, capability-enabled, and acting system.

This document owns the target-state boundaries for:
- distributed control-plane and execution-node operation
- governed learning, memory promotion, and procedure formation
- capability-scoped actuation
- operator-shaped automation
- enterprise-ready identity, authorization, and audit posture
- distributed execution target state
- learning, capability, and actuation target state

Not all elements described here are implemented yet. V8.2/B2+ is the active delivery frame, while the V8.1 living organization architecture remains the Soma-primary compatibility baseline that default operator surfaces must not regress.

## System Definition

Mycelis in V8.2 is a distributed AI Organization system that continuously operates, learns, and safely acts across environments under explicit governance.

```text
AI Organization
  -> Soma
  -> Team Leads
  -> Advisors
  -> Departments
    -> Specialist Roles
      -> Agent Instances
  -> Automations
  -> Reviews
  -> Activity
  -> Managed Exchange
  -> Learning and Context
  -> Procedures and Skills
  -> Routing
  -> Capabilities
  -> Distributed Execution Layer
```

## Operator Experience Rule

The default operator experience remains Soma-primary, intent-first, and simple by default.

advanced architecture/runtime control exists as a separate non-default layer for deep users. It must make inheritance, config origin, policy, approval posture, and runtime state legible without exposing raw internal execution detail as the main workflow.

## Managed Exchange

Managed exchange is the governed substrate between Soma, Team Leads, specialist roles, automations, MCP-backed systems, and distributed execution nodes.

Requirements:
- channels, typed fields, artifact schemas, and inter-agent threads become the governed exchange substrate between Soma, Team Leads, specialist roles, automations, MCP-backed systems, and distributed execution nodes
- channels, typed fields, artifact schemas, and inter-agent threads carry visibility, sensitivity, capability risk, and trust boundaries
- outputs are discoverable, indexable, and reviewable through structured exchange items, not raw transient message blobs
- semantic indexing supports continuity-aware retrieval such as related work and prior outputs
- managed exchange remains permissioned: visibility, sensitivity, capability risk, and trust boundaries travel with channels, threads, and artifacts rather than being inferred from transport alone
- normalization into exchange does not imply unrestricted trust
- external services, MCP tools, and future remote nodes remain governed by explicit trust classification and review policy

## Identity, Users, And Enterprise Auth

Authentication is pluggable; authorization remains internal and canonical.

External providers such as Entra ID, Google Workspace, GitHub, OIDC/OAuth, and SAML may prove user identity. Mycelis remains the source of truth for tenant roles, organization memberships, Soma access, capability permissions, approval authority, and audit visibility.

Edition posture:
- Free Node: single-user secure mode and local owner bootstrap
- Team: multi-user invitations, roles, and organization access
- Business: groups, service accounts, stronger admin controls, and audit visibility
- Enterprise: SSO/SAML/OIDC, Entra ID, Google Workspace, GitHub, SCIM target posture, domain verification, group-to-role mapping, policy enforcement, and advanced audit/export

Soma may use user context for communication and planning, but may not bypass authorization, approval, audit, memory visibility, or capability policy.

## Distributed Execution Layer

### Control Plane

Mycelis Core orchestrates tasks, manages organization state, controls policies, dispatches execution requests, and aggregates results. The primary host may be a local workstation, Raspberry Pi, or server-class deployment.

### Execution Nodes

Execution nodes run agent work close to the resources they need, such as GPUs, local filesystems, edge hardware, or cloud workers. Nodes return structured results to the control plane and must honor capability constraints.

### Communication Contract

```text
Control Plane -> Task Event -> Execution Node
Execution Node -> Result Event -> Control Plane
```

Requirements:
- reliable delivery
- idempotent execution
- failure recovery
- policy-checked capability scope
- persistent mission events for mutating actions

Each execution request includes organization context, role definition, task payload, and capability constraints. Each result includes output, status, metadata, and optional learning signals.

## Learning System

V8.2 activates governed learning loops:

```text
event/result
 -> review
 -> learning candidate
 -> classification + confidence + review posture
 -> evaluation
 -> promotion decision
 -> memory update
```

Promotion targets may include organization memory, team memory, role memory, and reflection/synthesis memory. Promotion requires policy validation, explicit memory-layer classification, confidence scoring, and operator approval when governance profile, capability risk, cost, or external-data thresholds require it.

Learning outputs may also form reusable procedures, role-specific behaviors, and automation improvements bound to specialist roles.

## Capability System

Agents may execute APIs, filesystem operations, browser automation, MCP tools, and hardware interfaces only through runtime capabilities.

Capability rules:
- deny by default and allowlist only
- scoped per role and execution
- auditable
- approval-aware for medium/high-risk usage
- never implied by template, prompt, or transport alone

## Automations

V8.2 makes automations editable while keeping them governed. Users can define triggers, scope, and purpose, but automations must not perform unsafe actions, hide execution, or bypass approval and audit rules.

Execution modes:
- interactive
- scheduled
- event-driven
- learning-driven
- distributed execution

## Safety Model

Core rules:
- no silent learning mutation
- no unrestricted actuation
- all capability usage is explicit, logged, and policy-checked
- approval gates and multi-step approval chains are available where risk requires them
- UI results normalize through the standard API envelope before operator consumption

## UI Evolution

The operator surface adds automation editing, approval and audit activity visibility, proposal cards, promotable learning insights, capability awareness, and clearer system liveness.

The UI still hides raw architecture and internal execution detail from the default workflow. Deep inspection belongs in advanced surfaces.

## Release Phases

1. Distributed execution: control plane, execution node, and basic task dispatch.
2. Learning promotion: memory promotion pipeline and controlled evolution.
3. Capabilities: safe execution layer and policy enforcement.
4. Automation editing: user-defined automations under governance.

## Final Principle

V8.2 transforms Mycelis from a structured AI Organization into a distributed, continuously learning, safely acting AI Organization system.
