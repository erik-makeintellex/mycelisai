# Mycelis V8.2 — Canonical Full Production Architecture
> Navigation: [Project README](../README.md) | [Docs Home](../docs/README.md)

## Purpose

This document is the canonical full production architecture for Mycelis.

V8.2 evolves Mycelis from a structured, observable AI Organization (V8.1) into a **distributed, learning, capability-enabled, and acting system** capable of:

* multi-host execution
* governed learning and memory promotion
* safe capability-based actuation
* operator-shaped automation
* continuous adaptive behavior

This document defines:
- the distributed execution target state
- the learning, capability, and actuation target state
- the production architecture and active B2+ delivery frame beyond the V8.1 baseline

Not all elements described here are implemented yet. V8.2/B2+ is the active delivery frame, while the V8.1 living organization architecture remains the Soma-primary compatibility baseline that default operator surfaces must not regress.

## Development Alignment

V8.2 is the full target.

Implementation progresses incrementally from the V8.1 compatibility baseline through bounded B2+ modules toward this architecture.

Documentation must always reflect:
- what is implemented
- what is planned

Alignment rule:
- `.state/V8_DEV_STATE.md` records what is actually complete, active, next, or blocked
- `README.md` summarizes the layered truth and points to the owning baseline/target docs
- V8.1 defines the compatibility baseline while V8.2 defines the active full target scope
- no silent divergence is allowed between implementation, release contract, and full architecture target

Operator surface rule:
- the default operator experience remains Soma-primary, intent-first, and simple by default
- advanced architecture/runtime control exists as a separate non-default layer for deep users
- that advanced layer must make inheritance and config origin legible without replacing runtime truth
- sensitive deployment secrets and host-specific wiring still remain file/env/config driven where appropriate

# 🧠 Core Architecture (V8.2 Extension)

```text
AI Organization
  -> Soma
    -> Team Leads
  -> Advisors
  -> Departments
    -> Specialist Roles
      -> Agent Instances

  -> Automations (Editable)
  -> Reviews (Evaluation + Learning)
  -> Activity (Events + Results)
  -> Managed Exchange
    -> Channels
    -> Threads
    -> Typed Artifacts / Messages
  -> Learning & Context (Semantic Continuity)
  -> Procedures / Skills
  -> Routing (AI Engine)
  -> Capabilities (Execution Permissions)

  -> Distributed Execution Layer
    -> Control Plane
    -> Execution Nodes
```

Managed exchange role in V8.2:
- channels, typed fields, artifact schemas, and inter-agent threads become the governed exchange substrate between Soma, Team Leads, specialist roles, automations, MCP-backed systems, and distributed execution nodes
- outputs must be discoverable, indexable, and reviewable through structured exchange items rather than raw transient message blobs
- semantic indexing over exchange items enables continuity-aware retrieval such as "find related work" and "find prior outputs for this task"
- managed exchange remains permissioned: visibility, sensitivity, capability risk, and trust boundaries travel with channels, threads, and artifacts rather than being inferred from transport alone
- normalization into exchange does not imply unrestricted trust; external services, MCP tools, and future remote nodes stay governed by explicit trust classification and review policy

# Identity, Users, And Enterprise Auth

V8.2 includes an enterprise-ready user-management and auth module while preserving the Free Node path.

Canonical module:
- [V8.2 User Management And Enterprise Auth Module](../docs/architecture-library/V8_2_USER_MANAGEMENT_AND_ENTERPRISE_AUTH_MODULE.md)

Core rule:
- authentication provider is pluggable
- authorization remains internal and canonical

External systems such as Entra ID, Google Workspace, GitHub, OIDC/OAuth providers, and SAML identity providers may prove who the user is. Mycelis remains the source of truth for tenant roles, organization memberships, Soma access, capability permissions, approval authority, and audit visibility.

Edition posture:
- Free Node keeps simple single-user secure mode and local owner bootstrap
- Team adds multi-user invitations, roles, and organization access
- Business adds groups, service accounts, stronger admin controls, and audit visibility
- Enterprise adds SSO, SAML, OIDC/OAuth, Entra ID, Google Workspace, GitHub, SCIM target posture, domain verification, group-to-role mapping, policy enforcement, and advanced audit/export

Soma must always operate inside policy:
- Soma can know the current user, role, allowed organizations, preferences, decision authority, and capability permissions
- Soma may use user context for communication and planning
- Soma may not bypass authorization, approval, audit, memory visibility, or capability policy

# 🌐 Distributed Execution Layer

## 1. Control Plane (Mycelis Core)

Responsibilities:

* orchestrates tasks
* manages organization state
* controls policies
* dispatches execution requests
* aggregates results

Runs on:

* primary host (e.g. Raspberry Pi)

---

## 2. Execution Nodes

Remote execution environments.

Responsibilities:

* execute agent tasks
* access local resources (GPU, filesystem, hardware)
* return results to control plane

Examples:

* gaming PC (high compute)
* edge device (hardware interface)
* cloud worker

---

## 3. Communication Model

Uses event-based architecture:

```text
Control Plane → Task Event → Execution Node
Execution Node → Result Event → Control Plane
```

Requirements:

* reliable delivery
* idempotent execution
* failure recovery

---

## 4. Execution Contract

Each execution request includes:

* organization context
* role definition
* task payload
* capability constraints

Each result includes:

* output
* status
* metadata
* optional learning signals
