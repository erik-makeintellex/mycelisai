Status: ACTIVE
Last Updated: 2026-06-19
Purpose: Compact, shareable architecture delivery document for the current MVP-to-release path.

Executive Summary

Mycelis V8.3 is converging from architecture expansion into operational embodiment. The product target is simple: I tell Soma what I want; Soma safely directs execution; I can see active work, outputs, proof, and recovery; I can trust what happened later.

Mycelis should now be described as a Soma-centered Outcome Management Engine powered by governed execution. The architecture remains deep internally, but the default user experience must collapse around visible execution, durable outputs, proof, recovery, and operator trust.

The operator-facing acceptance lens is defined in the V8.3 Soma User Experience Contract: users work through Soma, deliverables matter more than responses, proof and recovery must be visible, and advanced Inspect is optional.

The product-manifestation review is defined in the V8.3 Product Manifestation Architecture Review: every subsystem must justify user value, visibility, MVP classification, and adoption risk before it expands the visible product surface.

Product Architecture Shape

Outcome need
-> Soma understanding
-> governed proposal or direct answer
-> approved work when mutation/execution is required
-> deliverables and active lanes
-> proof, recovery, or re-entry

The default UI should not expose topology first. Advanced runtime detail belongs behind Inspect; a user should not need Agent, MCP, NATS, Prompt, Context Window, Topology, Vector Store, Event Bus, Workflow, or Run vocabulary to complete the default path.

Trusted Outcome Journey

Ask
-> Understand
-> Approve
-> Execute
-> Deliver
-> Trust
-> Recover
-> Revisit

Subsystems matter only insofar as they strengthen the journey. The MVP is complete when a non-technical user can complete the path without understanding agents, MCP, workflows, runs, topology, or infrastructure.

The release gate is twofold: the Infrastructure Disappearance Index proves the default path works without infrastructure vocabulary, and the Outcome Ownership Index proves a returning user can identify active outcomes, delivered outcomes, incomplete outcomes, attention needed, trust state, and next step within 15 seconds.

-> Soma evaluates intent

-> deliverable package appears
-> proof/audit/recovery is visible

Delivery Ownership Model

Lane A - Soma Inbox And Review

Owns Dashboard, review work, current work, and operator attention. Delivers Work Inbox tabs, list-detail review, one primary action per row, and selected detail with outcome, output, trust, recovery, conversation/events, and Inspect.

Lane B - Output Package And Receipt

Owns durable outputs and run receipts. Delivers one shared output package card, Soma and Groups convergence, rendered generated HTML/app opening, folder/resources/proof access, and source/code as secondary detail.

Lane C - Resources Capability Catalog

Owns tools, MCP, search, media, filesystem, AI engines, and runtime capability posture. Delivers capability-first Resources, availability states, risk and approval posture, and repair paths.

Lane D - Runs, Proof, And Recovery

Owns run receipts, recovery queue, and advanced inspect. Delivers receipt-first run pages, timeline/proof/raw payload behind tabs, unified recovery language, and the first advanced run-map stepper.

Lane E - QA And Release Proof

Owns validation and release confidence through one MVP proof script: login, Soma ask, proposal, approve, active work, output package, proof, Resources re-entry, and degraded sample recovery.

P0.1 Current Implementation

The output-package standardization slice is the active release-convergence priority. The earlier shared language and test fixture work is supporting groundwork inside this slice, not a separate release priority.

  • Shared delivery-runtime language helper: interface/lib/deliveryRuntimeLanguage.ts
  • Shared test fixture builders: interface/__tests__/support/deliveryRuntimeFixtures.ts
  • Low-risk callers use shared language for output folder action states, local media recovery copy, compact review/failure copy, and Active Work state labels.
  • Shared output-package path/action helper: interface/lib/outputPackageModel.ts
  • Soma and Groups share package open, folder, and Resources labels.
  • Groups project packages expose Open in Resources beside file and folder actions.

Proof so far: focused output-package Vitest coverage and interface typecheck. Remaining before acceptance: headed browser proof opening a generated browser app/game from Soma and Groups, plus Resources folder re-entry proof.

P0.1 acceptance must satisfy the Soma UX contract: the operator should receive a retained deliverable package, not just a chat response or raw source dump.

P0.2 And P0.3 Next Proof

P0.2 brings the services up cleanly and proves runtime readiness. P0.3 proves the output package in a real headed browser. These are separate because a healthy stack is necessary but not sufficient; the user behavior must also work.

UX Architecture Rules

Default routes must answer what the user can do, what needs attention, and where the output or proof is. Complex surfaces should use bounded list-detail layouts, tabs inside selected detail, receipts before logs, output packages before raw tool output, one primary action per row/card, and Inspect for topology or raw payloads.

Recovery And Confidence Posture

Every degraded state should expose what failed, what remains trusted, what proof is invalid, what can continue safely, what requires retry, what requires operator attention, and what uncertainty now exists. Confidence provenance should be prepared through schemas and receipts, but not overbuilt before the MVP path works.

Release Acceptance Standard

A slice is not accepted unless it reports the operator problem solved, files changed, runtime/API contract touched or reviewed, docs changed, tests run, headed/browser proof when visible UI changed, and remaining trust/recovery gaps.

Decisive release question:
Can the user complete the Trusted Outcome Journey and trust the result later?
