# V8 Demo Product Strike Team Plan

> Status: ACTIVE
> Owner: Product Management / Delivery Coordination
> Last Updated: 2026-03-29
> Primary Goal: Make Mycelis legible as an obvious product for a highly technical partner/funder demo without reducing Soma capability or deleting advanced power.

---

## Mission

Mycelis must become easy to understand in minutes:

- create an AI Organization
- work with Soma
- see what changed
- approve risky actions
- understand what the system retained

The default product experience must feel like a governed AI operating environment, not a nerd tool, swarm console, or architecture lab.

This strike team exists to deliver that outcome directly and in priority order.

---

## Non-Negotiable Product Rules

1. Do not reduce what Soma can do.
2. Do not delete advanced capabilities simply because they are complex.
3. Keep advanced/runtime/operator depth available through advanced surfaces and documentation, not in the default story.
4. The default experience must explain value in user language, not implementation language.
5. Governance, continuity, and execution proof must remain visible selling points, not hidden internal mechanics.

---

## Priority Goal Set

### Priority 1: Obvious Product Framing

A new technical evaluator should understand in under three minutes:

- what Mycelis is
- what Soma does
- why AI Organizations matter
- why governed execution is safer than a generic agent tool

### Priority 2: Demo Credibility

A single golden-path demo must prove:

- direct planning value
- governed mutation
- visible recent activity
- continuity / retained knowledge
- trust and operator control

### Priority 3: Feature Preservation

Existing power must remain available:

- advanced routes
- governance detail
- memory / RAG depth
- runtime/configuration depth
- specialist/council power

But that depth must stop competing with the default product story.

### Priority 4: Delivery Discipline

Every slice must:

- preserve feature truth
- preserve tests
- preserve documentation alignment
- avoid copy drift between landing, workspace, docs, and demo narrative

---

## Product Manager Role

The Product Manager for this strike team is the coordination authority for:

- scope control
- priority order
- team alignment
- acceptance criteria
- release-story coherence
- final go / no-go recommendation for demo readiness

The Product Manager must reject work that:

- improves implementation detail but weakens product legibility
- adds new default UI complexity without strengthening the product story
- hides governance or continuity in ways that reduce trust
- removes capability rather than relocating complexity appropriately

## Strike Team Structure

## 1. Product Narrative Team

### Purpose

Turn Mycelis into an obvious product story for technical partners, funders, and early adopters.

### Required mindset

- strong product framing
- high sensitivity to confusing language
- able to distinguish product value from implementation detail

### Primary deliverables

- clear homepage and first-run narrative
- consistent product language across default flows
- concise value framing for partner demos
- clear explanation of why Mycelis is different from generic agent tools

### Immediate focus

- landing page message hierarchy
- default CTA language
- organization creation framing
- value-language consistency across workspace panels

## 2. Default Experience Team

### Purpose

Make the default AI Organization and Soma experience feel simple, trustworthy, and strong.

### Required mindset

- ruthless focus on default workflow
- protects clarity over feature sprawl
- understands operator trust and action confidence

### Primary deliverables

- clean default navigation
- strong empty/loading/error states
- clear panel hierarchy
- readable causal and approval surfaces
- obvious first-step workflow inside the product

### Immediate focus

- AI Organization entry
- organization home
- Soma-first workspace
- recent activity / approvals / retained knowledge visibility

## 3. Capability Preservation Team

### Purpose

Ensure simplification never becomes accidental feature regression.

### Required mindset

- deep platform understanding
- skeptical of copy-only product work that might hide or orphan real capabilities
- strong dependency review discipline

### Primary deliverables

- feature retention audit
- advanced-surface retention map
- capability relocation plan where default UX is simplified
- proof that power features remain reachable

### Immediate focus

- advanced routes and docs reachability
- governance depth retention
- council/specialist access retention
- memory/RAG and configuration surface retention

## 4. Memory, Continuity, and Trust Team

### Purpose

Make retained knowledge, temporary continuity, and governance intelligible and trustworthy in product terms.

### Required mindset

- protects trust boundaries

### Primary deliverables

- default wording for durable memory vs temporary continuity

- continuity story that helps demos instead of confusing them
- operator explanation for “what was retained” and “why”

### Immediate focus

- Memory page summary language
- organization retention panel language
- governed proposal copy
- docs alignment for continuity and approvals

## 5. Demo Scenario Team

Build the single strongest golden-path story for partner/funder review.

### Primary deliverables

- one canonical prompt sequence

- choose the best first planning action
- choose the best governed mutation example
- choose the best continuity example

## 6. UI Testing Agentry Team

### Immediate focus

Ensure the demo/product story survives real startup, reset, and proof conditions.

## Team Synchronization Contract

Every team must update:

- what changed
- what user-facing effect changed
- what could regress elsewhere
- what docs/tests were updated
- what acceptance evidence now exists

No team may ship a “local improvement” without checking:

- product framing impact
- workflow impact
- feature-preservation impact
- docs/state impact
- demo impact

---

## Required Update Rhythm

### Product Manager receives:

- a start-of-slice goal statement
- an end-of-slice evidence report
- an explicit note about cross-team impact

### Teams share with each other:

- copy changes that affect workflow expectations
- route or panel changes that affect QA coverage
- retention/governance wording changes that affect docs and demo script
- feature relocation changes that affect advanced navigation or documentation
