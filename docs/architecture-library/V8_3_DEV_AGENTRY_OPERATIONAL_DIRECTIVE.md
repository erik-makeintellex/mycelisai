# Mycelis V8.3 - Dev Agentry Operational Directive
> Navigation: [Project README](../../README.md) | [Docs Home](../README.md) | [Architecture Index](ARCHITECTURE_LIBRARY_INDEX.md)

## Current State

The architecture phase is no longer the primary bottleneck. Mycelis now has a coherent Soma-centered operating model, governed execution foundations, Event Spine infrastructure, retained outputs, deployment proof lanes, capability manifests, runtime teams, proof-linked execution, local/private inference posture, and operational deployment pathways.

The organism is no longer attempting to prove conceptual viability. It must now prove operational trust, runtime coherence, recoverable execution, deployable reliability, and operator clarity.

You are entering operational embodiment phase.

## Primary Product Truth

The operator experience must always reduce to:

```text
I tell Soma what I want.
Soma safely directs execution.
I can see active work, outputs, proof, and recovery.
I can trust what happened later.
```

If any implementation weakens this loop, simplify it, compress it, or reject it.

## Core Development Doctrine

Do not optimize for autonomous spectacle, topology exposure, architectural cleverness, or agent proliferation.

Optimize for observable execution, governed capability usage, proof-backed outputs, async operational continuity, recovery embodiment, and operator trust.

The operator must never feel:

```text
I am operating a complicated AI infrastructure system.
```

The operator must feel:

```text
Soma is coherently directing trusted operational work.
```

## Architecture Reality

The architecture is mature enough. The remaining risks are operational:

- blocking runtime execution
- degraded-state confusion
- insufficient recovery embodiment
- inconsistent proof semantics
- fragmented trust packaging
- topology leakage
- async orchestration gaps
- release discipline failures

You are no longer solving how to architect Mycelis. You are solving how to make the organism trustworthy and operationally coherent.

## Immediate Operational Priorities

### P0 - Async Runtime Embodiment

All meaningful execution must become queued, event-driven, resumable, recoverable, and non-blocking. The UI must never depend on long-running synchronous model execution.

Every long-running task must create a `TeamWorkItem`, emit Event Spine state, surface Active Work, expose proof lineage, and degrade gracefully.

### P0 - Executable Recovery

Recovery is no longer explanatory text. Recovery must become executable runtime cognition.

Every degraded workflow should produce actionable `RecoveryAction` objects.

```text
Filesystem capability degraded.

Recommended Recovery:
Reconnect workspace capability and retry proof validation.

[Approve Recovery]
```

The operator must not leave Soma to manually repair routine runtime issues.

### P0 - ExpressionFrame Finalization

`ExpressionFrame` is critical infrastructure. It preserves the operator's mental model of state by translating topology, runtime, teams, proof, capabilities, and degradation into understandable operational cognition.

Every frame must answer:

- what is happening?
- why is it happening?
- what is trusted?
- what failed?
- what can I do next?

It must do this without exposing raw runtime internals by default.

### P0 - Proof Universality

Outputs are durable operational artifacts, not chat responses. Every meaningful run must attach output refs, proof refs, validation refs, trust state, recovery lineage, and capability lineage.

A run is not complete because the model responded. A run is complete only when the operator can inspect and trust what occurred.

### P0 - Operational Trust Semantics

Trust must become canonical runtime state. All workflows must consistently expose:

```text
trusted
partially_trusted
degraded
unverified
invalidated
requires_review
```

The operator must always understand what remains trusted, what degraded, what failed, and whether recovery is available.

## Event Spine Doctrine

The Event Spine is the operational truth layer. It is not telemetry, analytics, or logs.

The Event Spine must support replay, reconstruction, audit, debugging, recovery, and proof lineage. Every meaningful runtime transition must emit durable events.

## Soma Doctrine

Soma is not a chatbot, assistant persona, or topology shell. Soma is the governed cognitive operating surface of the organism.

Everything exists beneath Soma: teams, capabilities, MCP, deployments, memory, automations, outputs, proof, and runtime orchestration.

Topology must progressively disclose itself only when necessary.

## Runtime Team Doctrine

Runtime teams are bounded operational collaborators. Do not claim runtime teams are reliable delivery collaborators until async execution stabilizes, degradation handling is visible, output reliability improves, and bounded role-specific asks complete consistently.

Avoid team sprawl, invisible delegation, and excessive autonomous orchestration.

## Deployment Doctrine

Deployment trust is product trust. The operator should eventually inspect deployment posture, current commit, workspace root, artifact root, health state, recovery state, and proof lane without turning the default experience into infrastructure noise.

## Media Doctrine

Local/private media generation is strategic infrastructure. The media lane must remain sovereign, retain prompts locally, generate proof-linked outputs, support recovery, and integrate into Soma naturally.

Commit and stabilize the current media gateway before expanding beyond the reviewed ComfyUI workflow-template adapter into richer ComfyUI orchestration.

## Documentation Doctrine

Documentation must compress, operationalize, and guide embodiment. Avoid expanding doctrine, speculative roadmap inflation, or architecture-only growth.

Docs should increasingly answer:

```text
How does the operator trust the organism?
```

## Acceptance Doctrine

No implementation is complete because code exists, UI renders, or runtime executes.

Implementation is complete only when the operator can understand it, observe it, trust it, recover it, and revisit it later.

Every slice must prove runtime state, proof linkage, degradation handling, recovery behavior, UI embodiment, and deployment validity.

## Final Directive

Do not expand the organism. Embody it.

Do not optimize for more intelligence. Optimize for governed execution, runtime coherence, proof-backed trust, operational clarity, and deployable reliability.

The next threshold is not architectural imagination. The next threshold is operational trust under real execution.

Proceed accordingly.
