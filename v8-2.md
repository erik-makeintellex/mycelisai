# Mycelis V8.2 — Canonical Full Production Architecture

## Purpose

This document is the canonical full production architecture for Mycelis.

V8.2 evolves Mycelis from a structured, observable AI Organization (V8.1) into a **distributed, learning, capability-enabled, and acting system** capable of:

* multi-host execution
* governed learning and memory promotion
* safe capability-based actuation
* operator-shaped automation
* continuous adaptive behavior

This document defines:
- the distributed execution target state
- the learning, capability, and actuation target state
- the production architecture beyond the current V8.1 release

Not all elements described here are implemented yet. V8.1 remains the current bounded release target while V8.2 defines the full production target beyond that release.

---

# 🎯 System Definition

Mycelis in V8.2 is:

> A distributed AI Organization system that continuously operates, learns, and safely acts across environments under explicit governance.

---

# 🧠 Core Architecture (V8.2 Extension)

```text
AI Organization
  -> Team Lead (Soma)
  -> Advisors
  -> Departments
    -> Specialist Roles
      -> Agent Instances

  -> Automations (Editable)
  -> Reviews (Evaluation + Learning)
  -> Activity (Events + Results)
  -> Learning & Context (Semantic Continuity)
  -> Procedures / Skills
  -> Routing (AI Engine)
  -> Capabilities (Execution Permissions)

  -> Distributed Execution Layer
    -> Control Plane
    -> Execution Nodes
```

---

# 🌐 Distributed Execution Layer

## 1. Control Plane (Mycelis Core)

Responsibilities:

* orchestrates tasks
* manages organization state
* controls policies
* dispatches execution requests
* aggregates results

Runs on:

* primary host (e.g. Raspberry Pi)

---

## 2. Execution Nodes

Remote execution environments.

Responsibilities:

* execute agent tasks
* access local resources (GPU, filesystem, hardware)
* return results to control plane

Examples:

* gaming PC (high compute)
* edge device (hardware interface)
* cloud worker

---

## 3. Communication Model

Uses event-based architecture:

```text
Control Plane → Task Event → Execution Node
Execution Node → Result Event → Control Plane
```

Requirements:

* reliable delivery
* idempotent execution
* failure recovery

---

## 4. Execution Contract

Each execution request includes:

* organization context
* role definition
* task payload
* capability constraints

Each result includes:

* output
* status
* metadata
* optional learning signals
